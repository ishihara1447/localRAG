# 文抽出クッション フェア再検証（2026-07-15 → 2026-07-16 訂正・確定）

対象: `docs/RAG_EVAL_INTERNAL_AUDIT_2026-07-16.md` §6・§7優先度4「文抽出クッションをtemperature条件を揃えて再検証」への対応。

> **【重要な訂正 2026-07-16】** 本レポートの初版（07-15）は「temperatureを揃えると文抽出クッションの改善効果は確認できない（baseline 21/21 vs cushion 20/21）」と結論していたが、**これは誤りだった**。原因は、稼働中コンテナ（image `localrag-anythingllm:1.0.5`, 2026-07-12ビルド）の`lance/index.js`が**2026-06-29時点の古いコード**で、ハイブリッド検索も文抽出クッションも**実装ごと存在せず、一度も動いていなかった**こと。初版の「baseline」「cushion」はどちらも同じ古いdense-onlyコードを測っていたため差が出なかった。最新コードをコンテナに反映して測り直した確定結果を以下に記す。

## 背景

`scratchpad/cushion_hakusho30.py` によるstandalone検証は「baseline 19/30 → cushion 29/30」という劇的な改善を報告していたが、内部監査で baseline側temp=0.7・cushion側temp=0 という条件不一致（交絡）が発覚した。これを是正するため、本番実装（`server/utils/vectorDbProviders/lance/sentenceCushion.js`）を`LANCE_SENTENCE_CUSHION`のON/OFFで直接A/Bする再検証を行った。

## 検証環境の欠陥（初版が誤った原因）

再検証の途中で、**稼働コンテナのコードがホストの最新ソースより大幅に古い**ことが判明した。

| ファイル | コンテナ（image 1.0.5） | ホスト最新ソース |
|---|---|---|
| `lance/index.js` | 2026-06-29, 16,221 bytes, **hybrid痕跡ゼロ** | 2026-07-15, 23,900 bytes, hybrid+cushionフック入り |
| `lance/hybridSearch.js` | **存在しない** | 存在（2026-07-15） |
| `lance/sentenceCushion.js` | **存在しない** | 存在（2026-07-15） |

image `localrag-anythingllm:1.0.5` は2026-07-12ビルドで、ハイブリッド検索の実装（07-14〜15）・文抽出クッションの実装（それ以降）より前だった。さらに検証中に何度も `docker compose up -d` でコンテナを再作成しており、そのたびにイメージの古いコードに戻っていた。**結果として、このセッションで行った「hybrid ON」「cushion ON」の測定はすべて、実際には6/29のdense-onlyコードで実行されていた。**

### 是正方法

`docker compose up -d`（再作成）でイメージの古いコードに戻る問題を打ち消すため、再作成のたびに最新の `lance/index.js`・`hybridSearch.js`・`sentenceCushion.js`・`EmbeddingRerankers/native/index.js` を `docker cp` し `docker restart` するヘルパー（`scratchpad/apply_latest.sh`）を用意した。反映後、コンテナログで実際にクッションが発火することを確認した（`[NativeEmbeddingReranker] Reranking 37 documents...` / `[VectorDB::LanceDb] sentence-cushion chunks 8 -> 7, sentences kept 22/37`）。

## 正しい測定（すべて最新コード・`openAiTemp=0`固定・hybrid有効・各2回）

ワークスペース `hakusho-eval-57319043`（gemma4:12b, topN=8, 空白正規化済み1748チャンク）に対し、防衛白書30問（`hakusho_eval30.py`）を実行。

| 条件 | run1 | run2 | レンジ | a)事実 | b)数値 | **c)定義** | d)白書外 | e)言換 |
|---|:--:|:--:|:--:|:--:|:--:|:--:|:--:|:--:|
| baseline（cushion OFF） | 18 | 19 | **18〜19** | 6/8 | 4/6 | **2/6** | 4〜5/5 | 2/5 |
| cushion + int8（量子化, 既定） | 25 | 24 | **24〜25** | 6〜7/8 | 4/6 | **6/6** | 4/5 | 4/5 |
| cushion + fp16（非量子化） | 25 | 25 | **25** | 7/8 | 4/6 | **6/6** | 4/5 | 4/5 |

- int8 = 既定の `model_quantized.onnx`（570MB, リランキング約3.9秒/問）
- fp16 = HFの `onnx/model_fp16.onnx`（1.14GB, sha256照合済み, リランキング約9.3秒/問）を `model.onnx` として配置し `RERANKER_QUANTIZED=false` で読込
- 両者ともクッションの入力は同一（37文→22文採用）で、**リランカーの数値精度だけが異なる理想的な切り分け条件**

## 結論（確定）

1. **文抽出クッションは明確に有効**。baseline 18〜19 に対し cushion 24〜25 で **+6〜7ポイント**。2回ずつの実行でレンジが重ならない（18〜19 vs 24〜25）ため、単なるノイズでは説明できない。特に **c)定義説明カテゴリが 2/6 → 6/6（満点）** で、非核三原則・交戦権・文民統制など従来安定して落としていた設問がすべて正答。リランカーが定義文をピンポイントで最上位に持ち上げる効果が明瞭に出ている。

2. **量子化int8は効果を潰していなかった**。int8（24〜25）と fp16（25）は実質同等。当初「量子化ONNXリランカーの精度が悪くて効果を潰しているのでは」という切り分け仮説は**否定された**。**int8のまま採用でよい**（fp16はサイズ2倍・レイテンシ2.4倍で、今回の30問では見返りゼロ）。

3. **以前「クッション効果なし」と結論したのは検証環境の欠陥によるもの**であり、クッション技術自体は有効だった。ユーザーが当初提案した「ブースティング系モデル（リランカー）をワンクッション挟む」という着想は正しかった。

4. **統計的な厳密性の留保**（内部監査の作法に従う）: n=30での 18.5 vs 24.5（0.617 vs 0.817）は二項比率のz検定で両側p≈0.085（5%有意には僅かに届かず、10%水準では有意）。ただし①2回ずつのレンジが非重複、②c)カテゴリの2/6→6/6という質的に明瞭な改善、③失敗設問の顔ぶれが一貫、の3点から、**実運用上は有意な改善と判断してよい**。より確実にするなら評価セットを50〜200問へ拡張して再確認する（内部監査の中期課題）。

### トレードオフ（要注意・1件）

d)白書外（ハルシネーション検出）が baseline 5/5 → cushion 4/5 に低下。具体的には「自衛官が定年退職後に受け取る厚生年金の受給開始年齢」に対し cushion 時のみ「65歳です」と**捏造**する事象が int8・fp16 の両方・複数回で再現した。文を絞り込むことで文脈が短くなり、gemma4 が「不明」と言う代わりに一般知識で埋めてしまう副作用と見られる。+6〜7 に対し -1 なので正味プラスだが、**採用時はプロンプトで「絞り込んだ抜粋に該当が無ければ推測せず不明と答える」を再強調する**などの対策を検討すべき。

## 実務上の推奨

1. **文抽出クッションは採用価値がある**（int8リランカーのまま、既存の557MB同梱で追加コスト小、レイテンシ +約4秒/問）。ユーザーの当初アイデアが実測で裏付けられた。
2. **ただし採用には image の再ビルドが必須**（下記の重大課題）。現状の配布イメージ 1.0.5 には hybrid も cushion も入っていないため、compose の env を立てても実際には動かない。
3. fp16 は不要。`RERANKER_QUANTIZED` env と storage の fp16 `model.onnx` は検証用途のみ。配布物には含めない。
4. d)の捏造1件への対策（プロンプト強化）を採用とセットで行う。

## 派生した重大課題（本検証の副産物）

**配布イメージ `localrag-anythingllm:1.0.5`（および稼働中コンテナ）には、ハイブリッド検索も文抽出クッションも入っていない。** docs では「hybrid検索 既定ON確定（2026-07-15）」としているが、それはホストのソースコード上の話で、実際に動くビルド成果物には反映されていなかった。顧客配布前に、最新の `lance/` 一式（index.js / hybridSearch.js / sentenceCushion.js）と `EmbeddingRerankers/native/index.js` を含めて **image を再ビルドすること（WSL2 DNS問題があるため `docker build --network=host` 等の既知回避策を使う）** が必須。これは hybrid・cushion 双方の配布ブロッカー。

## その後の対応（2026-07-16 評価基盤の構造改善）

本再検証で判明した問題（cushion採用の可否、d)の捏造疑い、評価プロセスの脆弱性）を受け、以下を実施した。

1. **プロンプト強化（d)の数値捏造抑制）**: 「年齢・金額・料率・税率・年度などの具体的数値は、コンテキストに明示がある場合のみ回答し、一般に知られた数値でも記憶から補完しない」ルール8を追加（`fixtures/local/prompt-tuned.txt` と fork `server/models/systemSettings.js` の `saneDefaultSystemPrompt` 両方）。防衛白書30問で c)6/6 等の他カテゴリを一切壊さないことを確認（内部監査が警告した「v2で(d)を壊した」前例を回避）。

2. **d)設問の欠陥修正**: cushion有効時に「厚生年金の受給開始年齢」へ「65歳」と答える事象は**捏造ではなく評価セットの欠陥**と判明した。「厚生年金」は本文に0回だが「年金受給開始年齢である65歳」という記述が実在（2回, grep確認）し、cushionが正しく該当文を拾ってgemma4が忠実に答えた結果だった。この設問を、本文に完全に無縁な「雇用保険の失業給付（基本手当）の日額上限」（雇用保険・失業給付ともに本文0回, grep確認）へ差し替えた（`docs/RAG_HAKUSHO_EVAL_SET_2026-07-14.md`）。内部監査 §2-5「(d)の非存在立証が甘い」の的中例。

3. **評価ハーネスの移設・堅牢化**: scratchpad限定だった `hakusho_eval30.py` を git管理の `scripts/hakusho-eval.py` へ移設し、(a)**temperature=0固定**（決定性担保）、(b)**UNKNOWN判定の否定形要求**（裸の「含まれて」「記載されて」が肯定文にマッチしハルシネーションを見逃す脆弱性の除去）、(c)**(c)キーワード判定の空白正規化**（PDF字間空白による誤NGの再発防止）を実装。`scripts/scale-eval.py`（士業評価）にも同じUNKNOWN修正とtemperature=0固定を適用した。

4. **最終性能（確定）**: cushion + int8リランカー + プロンプトv2 + 設問修正で、防衛白書30問 **26/30 を2回とも完全再現**（temperature=0, a:7/8, b:4/6, c:**6/6**, d:**5/5**, e:4/5）。旧baseline(古コード)21から実質+5、cushion素の24〜25からプロンプト・設問修正でさらに安定化。残るNG（三文書決定日・防衛費の紛らわしい数値・営内隊舎の設問設計・発足日）はモデル/検索の実力または設問設計の限界で、採点バグではない。
