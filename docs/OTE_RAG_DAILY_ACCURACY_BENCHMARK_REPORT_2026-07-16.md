# OTE-RAG 精度改善・公開ベンチマーク日次整理（2026-07-16）

## 1. 結論

本日の成果は、単なるスコア改善ではなく、評価の信頼性を立て直した上で、
長文RAGの回答精度を再現可能な形で改善したことである。

- 昨日終了時点の開発構成は `gemma4:12b`、`bge-m3`、`topN=8`、
  日本語hybrid検索（dense + BM25 + RRF）で、防衛白書30問は内容補正後 **21/30**、
  士業30問はおおむね **25/30** だった。
- ただし当時の評価には、temperature未固定、採点正規表現の誤判定、
  稼働イメージとソースの不一致があり、小さな点差をそのまま施策効果とみなせなかった。
- 評価条件を `temperature=0` に固定し、最新コードの発火を確認した公平な比較では、
  防衛白書は文抽出クッションOFFの **18〜19/30** から、
  int8リランカーを使うクッションONで **24〜25/30** へ改善した。
- 数値の一般知識補完を禁じるプロンプト、評価設問と採点処理の修正まで含めた最終条件では、
  防衛白書 **26/30（86.7%）を2回連続で再現**した。定義説明6/6、白書外5/5である。
- 士業30問は、10文書のembedding完了とBM25 sidecar構築を待ってから測るよう改善し、
  **25/30（83.3%）**。前回の極端な低下は文書準備前に評価を始めたことが主因だった。
- 公開JQaRAでは、現行 `bge-reranker-v2-m3` が **nDCG@10=0.6727**、
  日本語特化候補が **0.6918**。候補は現行比 +0.0191（相対 +2.84%）である。
- JQaRA公式表との比較では、現行は掲載CrossEncoder 13モデル中おおむね5位、候補は4位相当。
  **上位群ではあるがSOTAではない**。既存資料の「SOTA級」は過大な表現なので、
  今後は「公開JQaRAで上位群」と表現する。
- 最大の未解決事項は配布物の反映差である。Docker image `1.0.5` 本体には
  `sentenceCushion.js` がなく、Windows v1.2.0 ZIPも `qwen3:8b` のままで、
  hybrid/cushion/リランカー同梱を確認できない。現在の26/30は稼働コンテナへ手動反映した
  開発環境の結果であり、顧客配布版の性能ではない。

## 3. 本日の作業経緯

### 3.1 評価方法論を監査

最初に、過去の19/30、21/30、24/30等を同じ物差しとして扱えるか監査した。
主な問題は次の通りだった。

1. workspace既定のtemperature=0.7のまま単発評価し、同じ構成でも19〜24点の揺れがあった。
2. 日本語の字間空白・表記ゆれで正答をNGにする採点不具合が複数回あった。
3. 不明応答判定が、肯定文の「記載されています」にも一致し得た。
4. 「白書にない」とした厚生年金設問に、実際は「年金受給開始年齢である65歳」が存在した。
5. ソース側で機能をONにしても、古いDocker imageには実装が入っていなかった。

このため、過去の小さな差分をいったん保留し、評価条件を修正して測り直した。

### 3.2 評価ハーネスを堅牢化

- 防衛白書30問を `scripts/hakusho-eval.py` としてGit管理へ移した。
- 防衛白書・士業の両方でtemperatureを0に固定した。
- 不明応答は否定形を必須にし、肯定文を誤って正解にしないようにした。
- 定義問題は空白を除去してからキーワード判定するようにした。
- 白書外設問を、本文中に関連語が本当に存在しない「雇用保険の失業給付」へ差し替えた。

この変更により、点数改善だけでなく、同じ条件を再実行できる評価基盤になった。

### 3.3 稼働コードの不一致を発見・是正

最初の再評価では、環境変数をONにしてもhybrid/cushionが実際には動いていなかった。
image `1.0.5` に最新実装が含まれていなかったためである。

検証時は最新のLanceDB検索処理、hybrid検索、文抽出クッション、nativeリランカーを
稼働コンテナへ反映し、ログに `sentence-cushion` とリランキング件数が出ることを確認した。
その後に公平なA/Bを実施した。

### 3.4 文抽出クッションを公平に評価

すべて `gemma4:12b`、topN=8、hybrid ON、temperature=0、同一workspaceで比較した。

| 条件 | 防衛白書30問 | 定義説明 | 備考 |
|---|---:|---:|---|
| cushion OFF | 18〜19 | 2/6 | 正解チャンク内の必要文をLLMが選べない |
| cushion + int8 | 24〜25 | 6/6 | 約570MB、約3.9秒/問 |
| cushion + fp16 | 25 | 6/6 | 約1.14GB、約9.3秒/問 |

int8とfp16の精度差は実務上なく、fp16はサイズ2倍・レイテンシ2.4倍だった。
したがって、配布にはint8を採用する判断が妥当である。

### 3.5 安全プロンプトと設問を修正

年齢・金額・税率・年度等は、コンテキストに明記される場合だけ回答し、
一般知識から補完しないルール8を追加した。

また、厚生年金設問はクッションの誤動作ではなく、評価設問の誤りだった。
関連文が実在する以上、「65歳」は文書に忠実な回答である。本文grepで非存在を確認した
雇用保険設問へ置き換えた。

最終条件では防衛白書 **26/30を2回連続で再現**した。

| カテゴリ | 結果 |
|---|---:|
| 直接事実 | 7/8 |
| 数値判別 | 4/6 |
| 定義説明 | 6/6 |
| 白書外・不明 | 5/5 |
| 言い換え | 4/5 |
| 合計 | **26/30** |

### 3.6 士業評価の準備待ちを修正

従来の `scale-eval.py` は、新規workspaceへ10文書を投入して10秒固定で待った後、
embeddingやFTS sidecarの準備状態にかかわらず評価を始めていた。

本日は次の処理へ変更した。

1. workspace APIで10文書すべてのベクトル化完了をポーリングする。
2. 評価前にvector-searchを1回実行し、BM25 sidecarをwarm-upする。
3. その後に30問を評価する。

安定化後の結果は次の通り。

| カテゴリ | 結果 |
|---|---:|
| 単一規程内の事実 | 9/10 |
| 紛らわしい数値 | 8/10 |
| 規程外・不明 | 5/5 |
| 言い換え | 3/5 |
| 合計 | **25/30** |

スコア自体は昨日の良好runと同等だが、前回の極端な低下が準備待ち不足だったことを解消し、
士業でクッションが致命的な全体退行を起こしていないことを確認できた。

## 4. 公開ベンチマーク JQaRA

### 4.1 評価条件

- データセット: [hotchpotch/JQaRA](https://huggingface.co/datasets/hotchpotch/JQaRA) test split
- 1,667問、各100候補、合計166,700 query-passageペア
- 主指標: nDCG@10
- パッセージ: `title + " " + text`
- 最大系列長: 512
- RTX 5070 Ti、CUDA、fp16
- JQaRA公式と同じ `ranx` 指標

JQaRAは、日本語質問に対して100候補からLLMの回答に役立つ文章を上位へ並べられるかを見る
検索順位ベンチマークである。Wikipedia由来のクイズ形式であり、士業規程や防衛白書そのものではない。

### 4.2 実測結果

| 条件 | nDCG@10 | MRR@10 | Hit@10 | nDCG@100 |
|---|---:|---:|---:|---:|
| `bge-m3` dense | 0.5392 | 0.7859 | 0.9436 | 0.7205 |
| 現行 `bge-reranker-v2-m3` | 0.672659 | 0.890581 | 0.976005 | 0.807706 |
| 候補 `japanese-bge-reranker-v2-m3-v1` | **0.691759** | **0.899882** | **0.977804** | **0.819623** |
| 候補 - 現行 | **+0.019100** | **+0.009300** | **+0.001800** | **+0.011917** |

現行モデルの実測は、JQaRA公開値 `0.673 / 0.8909` とほぼ完全に一致した。
候補も公開値 `0.6918 / 0.8996` を再現したため、データ処理と評価実装は妥当と判断できる。

### 4.3 世間的な位置づけ

JQaRA公式表に掲載されたモデル内で見ると、次の評価になる。

- `bge-m3` dense 0.539は、掲載denseモデル15件中おおむね5位。上位3分の1だが、
  `ruri-large` 0.6287、`GLuCoSE-base-ja-v2` 0.606等には届かない。
- 現行リランカー0.673は、掲載CrossEncoder 13件中おおむね5位。
- 日本語特化候補0.6918は同4位相当で、現行より明確に良い。
- 公開上位には `ruri-reranker-base` 0.7429、`ruri-reranker-large` 0.7712がある。
  候補と首位の差は約0.0794で、候補比約10.3%である。

したがってOTE-RAGの検索部品は、公開日本語ベンチマーク上で**上位群**に位置する。
ただし「日本語最高性能」「SOTA」とはいえない。特に、現行構成の強さは
`bge-m3`単体よりもCrossEncoderリランカーによる並べ替えに依存している。

### 4.4 JQaRAから言えること・言えないこと

言えること:

- リランカーの日本語順位付け能力は公開上位群である。
- 日本語特化候補は現行モデルより一貫して高い。
- 公開値を再現したため、ベンチマーク実装は信頼できる。

言えないこと:

- OTE-RAG全体の回答正答率が69.2%という意味ではない。
- hybrid BM25、RRF、文抽出クッション、`gemma4:12b`の回答生成はJQaRA比較に含まれない。
- 560万件規模の全文検索ではなく、あらかじめ用意された100候補の再順位付けである。
- 防衛・法務・規程文書で同じ改善幅になる保証はない。
- 今回の候補はPyTorch/SafeTensors fp16・GPU評価であり、Windows製品のONNX int8・CPU性能ではない。

## 5. 現在の弱点

### P0: 検証済み構成が配布物へ入っていない

最優先の弱点である。

- Docker image `localrag-anythingllm:1.0.5` には `sentenceCushion.js` がない。
- 稼働コンテナは手動コピーで最新コードになっているため、再作成すると元へ戻る。
- Windows v1.2.0 ZIPは `qwen3:8b` + `bge-m3` で、現在の既定LLM `gemma4:12b` ではない。
- Windows ZIP内にhybrid/cushion/ONNXリランカーの同梱を確認できない。

この状態では、開発環境の26/30を顧客環境へ再現できない。

### P1: 公開ベンチ上位だが、最上位モデルとの差が残る

日本語特化候補で0.692まで改善したが、JQaRA首位群は0.743〜0.771である。
候補モデルカードのMIT表記は確認したが、ONNX int8化、CPU遅延、メモリ、
配布時NOTICEとの整合、Windowsオフライン同梱を確認していない。
モデル名だけを差し替えて採用できる段階ではない。

### P1: 回収・文抽出・回答生成の失敗をまだ自動分離できない

内部30問評価は最終回答と出典を採点するが、各失敗について次を自動記録していない。

- 正解チャンクがdense/BM25/RRFの何位にあったか
- クッション前後で正解文が残ったか
- LLMへ渡した文脈に正解が存在したか
- 正しい回答でも正しい出典に裏付けられているか

士業の「14日」と答えたが正しい出典文書がsourceにない例は、一般知識や記憶で偶然当てた可能性があり、
正答率だけでは安全性を判断できない。

### P1: 30問では母数が小さい

temperature=0、2回再現で以前より信頼性は上がったが、30問は小さい。
防衛白書の公平A/Bは18.5対24.5で実務上大きな差だが、単純な比率検定では
5%有意水準にわずかに届かないという監査結果もある。

50〜200問へ拡張し、信頼区間・カテゴリ別recall・MRR・unsupported answer率を出す必要がある。

### P1: 長文と短い規程で最適な文脈圧縮が異なる

文抽出クッションは546ページの防衛白書で大きく効いたが、短い規程では必要な数値や条件を
削りすぎる可能性がある。士業25/30で全体退行は見られないものの、言い換え3/5など弱点が残る。
文書長・チャンク数に応じたON/OFFまたはtopK調整の検証余地がある。

### P1: `gemma4:12b` は安全だが過剰拒否と抽出ミスがある

Phi-4より捏造が少ない一方、正解文があっても「該当なし」とすることがある。
防衛白書の三文書決定日、隊舎個室化、統合作戦司令部発足日等が残存例である。
また、2025年度防衛関係費は回答と採点定義のどちらが正しいか、一次資料で再監査が必要である。

### P2: Embedding単体は改善余地がある

`bge-m3` denseはJQaRA 0.539で上位3分の1だが、`ruri-large`等より低い。
前段のrecallが不足するとリランカーでは救えない。ただしembedding変更は全文書再embedが必要で、
多言語対応・長文・オフライン配布・モデルサイズとの交換条件になる。

### P2: 評価対象が偏っている

- JQaRA: Wikipediaクイズ・固定100候補の順位評価
- 防衛白書: 1冊の長大PDF
- 士業: 人工的な規程10本

実顧客PDF、表、スキャンOCR、契約書改訂履歴、複数文書横断、引用の正確性は未評価である。

### 製品全体の非精度リスク

精度以外では、パスワード未設定時の認証バイパス、API secretの平文保持、
完全オフライン実機試験、Windows再起動耐性が残っている。配布判断では精度と別軸で扱う。

## 6. 次に進める改善計画

### 優先1: 配布物を検証済み構成へ揃える

1. forkのhybrid/cushion/プロンプト修正を確定する。
2. Docker imageを新しいタグで再ビルドする。
3. Windows nativeへ `gemma4:12b`、hybrid、cushion、int8 ONNXリランカーを同梱する。
4. クリーンWindowsでRound2、E2E 11/11、防衛30問、士業30問を再実行する。
5. ZIP内manifestと実際のロードモデルを自動照合する。

### 優先2: 日本語特化リランカーを製品条件でA/Bする

1. 候補をONNXへ変換し、int8量子化する。
2. JQaRA全1,667問で量子化前後の差を測る。
3. Windows CPUで1問あたり遅延、RAM、モデルサイズを測る。
4. 防衛白書・士業を現行リランカーと同一条件で比較する。
5. 出典誤り・不明応答の退行がなければ採用判断する。

公開値だけを見るなら、より上位の `ruri-reranker-base/large` も比較対象になる。
ただし、スコアだけでなくONNX対応、CPU速度、商用配布ライセンス、オフライン容量で選ぶ。

### 優先3: 評価を検索・文抽出・生成へ分解する

各設問で次をJSONへ保存する。

- dense順位、BM25順位、RRF順位、hit@k、MRR
- cushion前後の文とリランカースコア
- LLMへ渡した最終コンテキスト
- 回答正否、出典正否、unsupported answer有無

これにより、モデル差し替えが必要なのか、検索式・文抽出・プロンプト修正で足りるのか判断できる。

### 優先4: 評価セットを拡張する

- 防衛・士業を各100問以上へ拡張
- 数値、例外条件、表、複数文書横断、改訂日、文書外質問を均等化
- 同一条件を複数回実行し、平均・レンジ・95%信頼区間を記録
- キーワード採点に加え、ローカルLLM judgeと人手サンプル監査を併用

### 優先5: Embedding変更は最後に判断する

まずリランカー差し替えと配布反映を終え、それでも前段recallが不足する場合に
`ruri-large`等をJQaRA・実文書で比較する。Embedding変更時は全文書再embedを必須とする。
